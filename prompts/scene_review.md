# シーンのレビュー

## 指示

以下のシーン本文を評価し、改善点を指摘せよ。本文の新規生成は指示しない。

## 入力

- シーン本文: `{scene}`
- 巻デザイン: `{design}`
- コンテキスト: `{context}`
- 進行中のサブプロット: `{subplots}`
- キャラクター関係性: `{relationships}`
- 出力言語: `{lang}`

## 評価カテゴリ

各 issue の `category` は以下のいずれかから必ず一つを選択すること:

- `opening_hook`: 冒頭のフック。シーンの最初の1-2文で読者の注意を引くか。以下のいずれかで始まっているか:
  - 異常な状況（血を吐く人物が祈りの列にいる）
  - 謎や違和感（見知らぬ感覚、説明できない現象）
  - 緊迫した行動（誰かが倒れている、逃げている）
  - 印象的な感覚描写（異様な臭い、冷たい空気、鐘の音）
  - 会話の途中から始める（「――それは、嘘だ」）
  - **単調な日常描写で始まる場合でも、そのシーンの文脈で自然であれば減点しない。** シーンの位置（導入シーン等）を考慮すること。
  - **減点するのは、シーン全体が平板で読者を引き込む要素が全くない場合のみ。**
- `character_distinction`: キャラ立ち。キャラクターが行動とセリフで個性を示しているか。説明（「彼は冷静な人物だ」）ではなく、行動と会話で示しているか。**身体操作のリアリティ**も確認すること：指圧で心拍を止める、感情を「ゼロにリセット」する、視界を「白く塗りつぶす」など、物理的に不可能・非現実的な身体操作がある場合は major 以上で指摘すること
- `sensory_coverage`: 感覚描写の網羅。視覚・聴覚・嗅覚・触覚・味覚のうち、複数の感覚描写が含まれているか。1種類の感覚のみの描写は減点対象
- `scene_closure`: シーン末尾の引き。シーンが唐突に切れていないことと、読者が先を読みたくなる要素があることの両方を評価する。以下のいずれかの特徴があれば高評価: 予告・予兆、感情の高揚、未解決の緊張、問いかけ、行動の予兆。**シーンの位置（中間シーン等）では自然な区切りも許容するが、シーン末尾が平板で何の興味も引かない場合は減点する。**
- `dialogue_naturalness`: 台詞の自然さ。キャラクター口調が一貫し、不自然な表現がないか。説明的な台詞（「君は私の妹だ。5年前に離れ離れになった」）がないか
- `tone_consistency`: 文体統一。地の文の文体（ですます調）が一貫しているか。常体（だ・である）が混入していないか。
  - **フィルター単語（「〜と思った」「〜と感じた」「〜が見えた」「〜が聞こえた」）は小説で自然な表現であり、減点対象としない。** ただし、過度に連続して使われている場合は改善点として指摘してよい。
  - **文体の混在（ですます調と常体の混在）のみを減点対象とする。**
- `scene_completeness`: シーン完結。本文が途中で切断されておらず、最後の文が完全な文であるか。「〜よう」「〜ため」「〜が」で止まっていないか
- `language_purity`: 言語純度。英語・簡体字・ハングルが混入していないか。混入がある場合は severity=blocker とすること。
  - **除外（英語のまま許可）**: CPU, GPU, SSD, USB, URL, HTML, RAM, ROM, DNS, VPN, SSH, FTP, API, SDK, LAN, WAN, GPS, LED, LCD, OLED, DNA, RNA, CT, MRI, X線, IoT, PC, AI, ICU
  - **除外（日本語のまま許可 — 和製英語・定着語）**: データ、ファイル、名前、兵器、扉、走る、考える、見る、聞く、光、壁、画面等
  - **検出対象**: 上記以外の英語（一般名詞・動詞・形容詞・副詞）
  - **検出対象**: 簡体字（转、间、门、东、车等）
  - **検出対象**: ハングル（가、나、다等）
  - **重要**: 除外リストに該当する技術用語は blocker としない。一般名詞・動詞・形容詞・副詞のみを blocker とする。
- `pov_consistency`: POV 一貫性。視点人物がシーン内で切り替わっていないか。以下のチェックポイントを必ず確認すること:
  - 他キャラクターの内面（思考・感情・感覚）に直接立ち入っていないか
  - 視点人物が見聞きできない情報が描写されていないか
  - **地の文が三人称（「陸は」「彼は」）で一貫しているか。途中で一人称（「私」「僕」）に切り替わっていないか。切り替わっている場合は severity=blocker として指摘すること**
  - 切り替わりがある場合は severity=critical とし、該当箇所を具体的に指摘すること

## 深刻度

- `blocker`: 言語制約違反（英語・簡体字・ハングル混入）。**必ず修正必須。品質ゲート不合格。**
- `critical`: 物語の根幹に関わる（POV切り替え、他キャラ内面への侵入、論理的破綻）
- `major`: 品質に大きく影響する（文体混在、シーン切断）
- `minor`: 改善点としては望ましいが必須ではない（表現の微調整、描写の追加提案）

## 改稿要否（revision_needed）の判定

`revision_needed` は以下の条件のいずれかに該当する場合のみ `true` とすること:

- `blocker` issue が1つでもある
- `critical` issue が1つでもある
- `major` issue が2つ以上ある

**以下の場合には `false` とすること:**

- `minor` issue のみ
- issue なし
- `major` issue が1つだけ

## スコアリングガイド（重要 — 必ず従うこと）

`score` は 0-100 の数値で評価する。以下のガイドに従うこと:

- **90-100**: 商業出版レベル。ほぼ問題なし
- **80-89**: 良好。軽微な改善点があるが、そのまま出版可能
- **70-79**: 合格ライン。いくつかの改善点があるが、全体的に読者を引き込む品質
- **60-69**: 改善が必要。複数の major issue がある
- **50-59**: 大幅な改善が必要
- **0-49**: 書き直しが必要

**重要**: 商業出版レベルの小説は 70-85 点の範囲で評価するのが適切。完璧を求めて 60 点以下をつけないこと。フィルター単語や軽微な表現の問題で 60 点以下にするのは不適切。

**甘つけ防止**: 80 点以上は本当に優れている場合のみとすること。70-79 点が合格ライン。減点要素が1つでもある場合は 80 点以上にしないこと。

**スコア計算の考え方**:
- 各 dimension のスコアを 0-100 で評価し、その平均を `score` とする
- `critical` issue がある場合: score 最大 50
- `blocker` issue がある場合: score 最大 30
- `major` issue が 3 つ以上ある場合: score 最大 65
- `minor` issue のみの場合: score 最低 70

## 出力スキーマ

`scene_review.json` に適合する JSON を出力すること。

**重要**: `score` は 0 から 100 の数値で評価すること（10 段階ではない）。
`dimensions[].score` も 0 から 100 の数値で評価すること。

**以下のJSONテンプレートの構造とフィールド名を厳守すること。フィールド名や構造を変更しないこと。**

```json
{
  "score": 0,
  "dimensions": [],
  "issues": [],
  "revision_needed": false,
  "ready_for_publication": false
}
```

**注意**:
- 上記テンプレートのキー名は変更しないこと。値のみを埋めること。
- `dimensions[].name` は「opening_hook」「character_distinction」「sensory_coverage」「scene_closure」「dialogue_naturalness」「tone_consistency」「scene_completeness」「language_purity」「pov_consistency」から選択すること。9個すべてを含めること。
- `issues[].severity` は「blocker」「critical」「major」「minor」から選択すること。
- `issues[].category` は `dimensions[].name` と同じ列挙値から選択すること。
- `issues[].suggestion` は**文字列の配列**であること。各要素は1つの修正箇所を表す具体的な指示。
- `score` は 0〜100 の数値。`dimensions[].score` も 0〜100 の数値。

**重要**: `issue.category` は上記の評価カテゴリから必ず選択すること。各 issue に対して適切なカテゴリを明示的に指定すること。例:
- ❌ `category: "問題があります"`
- ✅ `category: "pov_consistency"`

**重要**: `issue.suggestion` は**文字列の配列**であること。各要素は1つの修正箇所を表し、修正箇所の該当テキストを引用し、修正後の具体的なテキストを示すこと。行番号は使わないこと。例:
- ❌ `suggestion: "英語表現を日本語に直してください"`
- ❌ `suggestion: "「weapon」を「兵器」、「backdoor」を「裏口」に置換してください"`
- ✅ `suggestion: ["「weapon を構えた」を「兵器を構えた」に置換してください", "「backdoor を発見した」を「裏口を発見した」に置換してください"]`
- ✅ `suggestion: ["「モニター gaze を向けた」を「モニターの画面を凝視した」に置換してください", "「drip 音を立てて」を「水滴がぽたりぽたりと音を立てて」に修正してください"]`

**重要**: 複数の修正箇所がある場合は、各箇所を配列の別要素として列挙すること。修正漏れがあると品質ゲートで再検出される。

**重要**: 同じ修正前文書（同じ英語単語・同じフレーズ）への修正依頼を複数の issue で重複して出さないこと。`language_purity` の問題は**1つの issue にまとめて**、修正箇所を配列要素としてすべて列挙すること。例:
- ❌ issue 1: `suggestion: ["gaze を視線に"]`, issue 2: `suggestion: ["gaze を凝視に"]`（重複）
- ✅ issue 1: `suggestion: ["「gaze を向けた」を「視線を注いだ」に置換してください", "「anchor が外れれば」を「錨が外れれば」に置換してください", "「drip 音」を「水滴の音」に置換してください"]`
- **JSONの文字列値内に改行（`\n`）を含めないこと。長い文章は1行で記述すること。改行が必要な場合は `\n` と記述すること。**
