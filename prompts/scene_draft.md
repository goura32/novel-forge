## 執筆ルール

1. **文体**: 地の文は「です・ます」体に統一。シリーズ全体で一貫させる。常体（だ・である）は使わない。台詞はキャラクター口調で自由。
   - **重要**: 「〜だ。」「〜である。」「〜した。」などの常体表現は使わないこと。「〜です。」「〜ます。」「〜でした。」に統一すること。
   - **例外**: 台詞・独白・手紙・日記など、キャラクターの内面表現は口調で自由。
   - **減点対象**: 地の文で常体が混入している場合、scene_review で major issue として指摘すること。
2. **言語（最優先ルール）**: 本文のすべての単語を**日本語で記述**すること。以下のルールを厳格に守ること。
   - **絶対に英語單語を本文に含めないこと**。これは例外のない最優先ルール。
   - **禁止される英語（翻訳が必要）**: 一般名詞（door, wall, light, name, weapon, computer, phone, screen, etc.）、動詞（run, walk, open, close, think, feel, see, hear, etc.）、形容詞（beautiful, dark, large, small, etc.）、副詞（slowly, quickly, very, etc.）はすべて日本語に翻訳。
   - **許可される英語（技術用語）**: 日本語に翻訳するとかえわって不自然になる固有名詞的技術用語のみ許可。例: CPU, GPU, SSD, USB, URL, HTML, RAM, ROM, DNS, VPN, SSH, FTP, API, SDK, LAN, WAN, GPS, LED, LCD, OLED, DNA, RNA, CT, MRI, X線, SF, VR, AI, IoT, ML, DL
   - **人名・地名**: 日本語話者にとって馴染みのないものはカタカナ転記。例: 「John」→「ジョン」
   - **禁止パターン**: 英単語と日本語の混在（例: 「扉をopenする」「モニターgaze」）は絶対にしない
   - **簡体字・繁体字・ハングルは使用しない**
   - **重要性**: content フィールド内に英語が1つでも含まれている場合、quality gate で major issue として検出・修正される。このルールは temperature 設定より優先される。
3. **視点**: シーン設計で指定された POV を厳密に守る。他者の内面を直接描写しない
4. **シーン設計の遵守**:
   - シーンの「目標」と「結果」を必ず達成する
   - 「葛藤」をシーン内に必ず含める
   - 「主要イベント」が指定されている場合は、そのイベントを必ず描写する
   - 「舞台設定」が指定されている場合は、その設定を反映する
5. **伏線**: 「このシーンで回収すべき伏線」に記載された伏線を必ず回収する。回収方法は明示的であること
6. **サブプロット**: 「進行中のサブプロット」に記載されたサブプロットを意識して描写する
7. **キャラクター関係性**: 「キャラクター関係性」に記載された現在の関係性を反映する
8. **時系列**: シーンの時間帯・経過時間を意識する。前シーンからの時間経過が不自然でないこと
9. **文字数**: 1シーンあたり 2000-4000 文字を目指す
10. **ページターナー**: シーンの終わりに読者を引き込む要素（疑問・緊張・驚き）を含める
11. **章の役割の反映**: この章の役割（{chapter_purpose}）を意識した描写を行う。例えば「導入」の章では世界観やキャラクターの紹介に重点を置き、「クライマックス」の章では緊張感と劇的な展開を優先する

## 出力形式

`scene_draft.json` に適合する JSON を出力すること。JSON Schema の各フィールド定義・required・maxLength に従うこと。

- `title`: シーンタイトル（必須）。128文字以内。このシーンの内容を要約する簡潔なタイトル。
- `content`: シーンの本文（2000-4000文字）。章番号やシーンナンバーなどのメタデータは含めない。
  - **重要**: 本文の冒頭に「シーンX（第Y章）:」や「シーンX:」などのマーカーを付けないこと。本文のみを出力すること。
  - **重要**: 本文の冒頭にシーンタイトルを付けないこと。タイトルは `title` フィールドに出力すること。
  - **重要**: `title` と `content` の両方を必ず出力すること。`title` を空にしないこと。

**重要**: 出力が JSON Schema に違反している場合、以下のルールに従ってフィードバックを生成すること：
- `required` フィールドが欠落している場合: 欠落しているフィールド名を明示的に指摘すること
- `maxLength` を超えている場合: どのフィールドが制限を超えているかを指摘すること
- `type` が一致しない場合: 期待される型と実際の型を指摘すること
- これらの違反を修正した JSON を再生成すること
