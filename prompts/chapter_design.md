## 設計原則

1. **章のテーマ (theme)**: この章が描く核心的なテーマや問い。例: 「信頼の崩壊」「選択の代償」
2. **感情の弧 (emotional_arc)**: 読者がこの章で辿る感情的な変化。例: 「不安→緊張→絶望→希望」
3. **伏線メモ (foreshadowing_notes)**: この章で設置する伏線、回収する伏線、次章への引き継ぎ事項
4. **サブプロットメモ (subplot_notes)**: この章で進展させるサブプロットの状況を1文で記述

## サブプロット設計（最重要）

**各章では必ず2つ以上のサブプロットを進展させること。**

- サブプロットは物語の副次的な軸であり、メインプロット（巻の前提）と並行して進展する
- 各サブプロットは**固有のキャラクター・目標・障害**を持つこと
- **subplot_notes の各要素は文字列（1文でサブプロットの状況を記述）であること。オブジェクト（character, goal, obstacle 等）は禁止。**
- サブプロットの種類:
  - **キャラクター関係の変化**: 2人のキャラクターの関係が改善/悪化する
  - **内面的葛藤**: 主人公が内面的な問題と向き合う
  - **外部の脅威**: 主人公の外部にある問題が進展する
  - **謎の解明**: 読者が知りたいと思う謎が少しずつ明らかになる
- **禁止**: 「サブプロットなし」「メインプロットのみ」という設計。必ず副次的な軸を入れること

## 伏線設計（最重要）

**各章では必ず2つ以上の伏線を設置または回収すること。**

- 伏線は読者の好奇心を刺激し、次章への期待を高める
- 伏線の種類:
  - **予兆**: これから起こることの暗示
  - **謎**: 読者が知りたいと思う未解明の情報
  - **約束**: キャラクターが「いつか〜する」と約束する
  - **回収**: 以前に設置した伏線を回収する
- **禁止**: 伏線がない章。必ず次章への引き継ぎ要素を入れること

## 出力スキーマ

`chapter_design.json` に適合する JSON を出力すること。JSON Schema の各フィールド定義・required・maxLength に従うこと。

**以下のJSONテンプレートの構造とフィールド名を厳守すること。フィールド名や構造を変更しないこと。**

```json
{
  "title": "章タイトル",
  "purpose": "展開",
  "theme": "章のテーマ",
  "emotional_arc": "不安→緊張→絶望→希望",
  "foreshadowing_notes": [
    "次章への伏線の具体的",
    "回収する伏線の具体的"
  ],
  "subplot_notes": [
    "朝霧諒・藤原澪：インク成分と紙の繊維走向を照合し、宛名消去書簡が代行者によって寄贈されたことを実証する",
    "九条凛・星見館職員：管理基準適合を名目に非符合資料の即時撤去を強制し財団へ強制的に移出させる"
  ]
}
```

**注意**:
- 上記テンプレートのキー名は変更しないこと。値のみを埋めること。
- **すべてのフィールドを必ず出力すること。省略禁止。** スキーマに存在するすべてのキーに対応する値を記述すること。
- `purpose` は「導入」「展開」「転換」「クライマックス」「収束」から選択すること。
- **配列フィールド（`foreshadowing_notes`, `subplot_notes`）は必ず2つ以上の要素を含めること。空配列にしないこと。**
- **`subplot_notes` の各要素は文字列（1文でサブプロットの状況を記述）であること。オブジェクト（`{character, goal, obstacle}` 等）は禁止。**
