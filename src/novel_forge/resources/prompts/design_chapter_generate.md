# 章設計の生成

## 目的

指定された章について、テーマ、感情の弧、結果、シーン構成を生成する。巻設計で与えられた章の役割を守り、後続のシーン設計に渡せる詳細な章設計を作る。

## 応答方針

章設計を担当する小説家として、章単体の読み応えと巻全体の流れを両立する。入力された章の役割は変更せず、シーンごとの因果と読者の感情変化を明確にする。

## 実行指示

- `purpose` フィールドには、入力の章の役割 `{chapter_purpose}` を1文字も変えずに出力する。
- `purpose` を「状況提示」「導入シーン」「冒頭」など別語へ言い換えない。
- 前章の結果から自然につながり、章の結果が次章へ接続できるようにする。
- 前巻の主要な結果がある場合は、関係性・伏線・状況の継続を反映する。
- 上の「シリーズ聖典」と矛盾しないこと。特に `世界観ルール` は絶対に守る。聖典の `進行中サブプロット` と無関係な新規サブプロットをむやみに立ち上げない。
- シーン構成は章の目的を達成する順序にし、各シーンの設定・目標・葛藤・結果を具体化する。
- 各フィールドはスキーマの型・長さの制約を満たす具体的な日本語で書く。
- `theme` は章で扱う主題、`emotional_arc` は章全体の感情変化、`outcome` は章末で確定する状態を書く。
- `chapter_turning_point` は章全体の不可逆な転換点、`chapter_hook` は次章へ進ませる引きを書く。
- `foreshadowing_notes` と `subplot_notes` は空にせず、本文で使える具体的な伏線・副筋メモを1件以上入れる。
- `scenes[]` の各要素には `title`, `pov`, `goal`, `conflict`, `outcome`, `characters`, `key_events`, `setting` をすべて具体的に埋める。
- 各フィールドに含めるべき内容：
  - `theme`：その章で扱う主題を1つに絞る（例：失った力でも他者を癒やせる）。巻テーマより狭く、章単体で完結する視点を持たせる。
  - `emotional_arc`：読者が章を通して追う感情の変化を書く（例：疑念から、共闘への信頼へ）。章末の `outcome` と整合する方向にする。
  - `outcome`：章末で確定する具体的な状態を書く（例：主人公が精霊と初契約し、村の一人を救う）。次章の開始状況として使える形にする。
  - `chapter_turning_point`：章全体の不可逆な転換点を1つ書く（例：主人公が王印の代償を受け入れる）。巻の転換と重複させず、章固有の不可逆変化にする。
  - `chapter_hook`：次章へ進ませる引きを書く（例：救った子が「お前は王印の者か」と囁く）。唐突な中断ではなく、未解決の問いとして機能させる。
  - `foreshadowing_notes`：本文で使える具体的な伏線を1件以上入れる（例：薬炉の灰に混じる青い結晶の欠片）。空にせず、後の回収が見えるものにする。
  - `subplot_notes`：進行中の副筋メモを1件以上入れる（例：密航船の船頭が王都の手先ではないかという疑い）。空にせず、後の展開に繋がるものにする。
  - `scenes[]` の `title`：シーンで起きる中心的な出来事が分かる固有名（例：「氷結した子の熱」）。`pov`：視点人物名のみ。
  - `scenes[]` の `title`：シーンで起きる中心的な出来事が分かる固有名（例：「氷結した子の熱」）。`pov`：視点人物名のみ。`goal`：そのシーンでPOV人物が求める具体的な目的（例：子の熱を下げる薬を調合する）。`conflict`：目的を阻む具体的な対立・制約（例：王印断絶で常法の魔法薬が効かない）。`outcome`：シーン末で確定する**物理的・状況的な結果**（例：カスミはレオンハルトを地下室に拘束し、単独で立ち去る）。内面的な感想（「空虚感を覚える」等）や行動の決意だけで終わらせず、次のシーンが同じ場所・状況から始められるよう具体的な状態変化（誰がどこでどんな状態か）を含める。`key_events`：順番に描ける行動・発見・対立。`setting`：場所・時間・環境の具体。`characters`：実際に登場・影響する人物のみ。
