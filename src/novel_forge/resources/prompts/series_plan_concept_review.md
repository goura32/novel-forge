# シリーズ構想のレビュー

## 目的

シリーズ構想を編集者視点で評価し、後続工程に進める前に修正すべき問題を特定する。問題がある場合は、改訂工程がそのまま適用できる具体的な差分を示す。

## 応答方針

企画の良し悪しを過剰に粗探しせず、商業性・明確性・一貫性を基準に必要な指摘だけを行う。改善案は抽象論ではなく、そのまま採用できる完成文にする。

## 実行指示

以下の観点を順に評価する。

### レビュー観点
- あらすじ: 「誰が、何に、どう立ち向かうか」が読み取れるか。
- ジャンル: ジャンル設定と内容が矛盾していないか。
- 世界観: 独自ルールが明確で、相互に矛盾していないか。
- ターゲット: 読者像が具体的か。
- 差別化: 入力キーワードから作っただけの凡庸な企画になっていないか。
- 言語純度: 日本語文脈で不自然な簡体字、中国語構文、英語混在、ハングル混在を指摘する。指摘時は問題文字列を引用する。自然なカタカナ語、英語表記、英字略語、一般的なジャンル語、固有名詞、日本語として成立する漢語は問題にしない。
- 中核ギミック: 入力キーワードまたはタイトルに中核ギミックが含まれる場合、何が、誰に、どの範囲で、どの条件で変化・交換・喪失・制限されるのかを確認する。別種のギミックへ置き換わっている場合はimportant 指摘にする。指摘の `after` でも元のギミックの型を保持し、別種のギミックへ変換しない。
- 内容欠落（各フィールドが持つべき内容が書かれているか）:
  - `world_rules[]` の各要素に、そのルールが存在する理由、発動時の効果、制約の範囲、例外や緩和の条件と代償の少なくとも3つが含まれていない場合はimportant 指摘にする。単なる名称や「〜というルールがある」だけの記述は不十分。
  - 喪失・剥奪・封印によって新たな力や能力が使える構造の場合、`world_rules[]` に「喪失が何を解放・触発し、なぜ別系統の力が発現するか」の因果の方向性が全く書かれていない（完全欠落）場合のみimportant 指摘にする。1文で因果の方向性が読めれば十分とし、物理的・魔術的な詳細メカニズムの完全な論理説明や、より詳しい因果の要求はしない。
  - `logline` に希望・克服の伏線を含める場合、それが「王印を取り戻す」「魔法を再び使えるようになる」など永久喪失と両立しない型であれば、別系統の力（精霊契約・植物媒介・独自療法）による確立へ書き換えるようimportant 指摘にする。ただし「王印に頼らない別系統の力の確立」「失われた権威ではなく自分の力で価値を見出す」型の伏線と、それに対応する別系統の力のルールがあれば矛盾とはみなさない（制約と葛藤は物語の前提であって矛盾ではない）。

### 指摘対象
- レビュー結果は、レビュー対象に根拠があり、改訂工程で具体的に修正できる問題に限定する。
- severity は後続工程への影響度で決める。必ず英語のみ使用：critical / important / minor の3値だけ。日本語severity（"致命的""重要""軽微"など）や別の表記は一切禁止。軽微に見える表記・用語でも、後続工程で固定化・拡散する場合は指摘する。一方、好み、根拠のない言い換え、さらなる具体化要求、既に成立している設定の別案は `issues` に含めない。
- 指摘は既存入力とスキーマに基づく修正可能な差分に限定し、説明だけの意見を含めない。
- 入力キーワード・前工程JSONと矛盾する性別、役割、固有名、因果関係、巻間伏線の未回収はimportant 以上として指摘する。
- 入力キーワードまたは前工程JSONに明示された期間、職業、役割、性別、タイトル、ジャンル、固有名は正として扱い、好み・もっともらしさ・別案のために変更提案しない。
- 後続工程で具体化できる未定義要素、単一フィールドだけでは説明しきれない背景、さらなる根拠付けの要望、人物の詳細設定、逃走・追跡ギミックの運用細部はimportant 指摘にしない。
- 同一JSON内で直接矛盾している場合だけ、既存値を最小変更で整合させる。世界観や人物設定を新規に発明して置き換えない。
- 物語の前提としての制約と葛藤は矛盾とみなさない（原則）:
  - 世界観に不利益な制約ルール（立入禁止、能力喪失、権限剥奪等）がある場合、それと主人公の background（元魔法使用者、元宮廷職、かつて力を持っていた等）や、logline の帰還・復讐・情報入手・葛藤要素は矛盾ではない。制約があるからこそ物語が成立する。
  - 「かつて使えたはずのものが使えなくなる」は背景設定と世界ルールの整合性ではなく、葛藤の表現である。矛盾として指摘してはならない。
  - 制約（立入禁止等）がある状態で、logline が「Xを知る鍵」「復讐」「帰還」を含むのは、主人公がその制約を打破・迂回しようとする物語の前提であり矛盾ではない。
  - logline の主人公焦点（例：「元皇女が～」）と selling_points や world_summary の人物呼称違い（例：「追放されたヒロイン」）は、同じ主体を異なる視角・文脈で言及しているだけであり矛盾しない。
- 複数の力系統の共存は矛盾とみなさない（原則）: 同一世界観内で複数の力系統（王印由来／精霊契約／血統／技術等）が共存し、それぞれ別のルールで説明されていれば矛盾ではない。ただし、**同じ条件・同じ系統で**「使える」と「使えない」が並列に存在する場合のみ指摘する。
- 既に因果関係が読み取れる `logline` や `world_rules` を、明確化・文体改善・もっと精密そうな別案という理由だけで指摘しない。特に「喪失・剥奪・封印」によって別系統の力が得られる構造では、「[喪失]によって[従来の機能/制約]が失われ、その結果[別系統の力]が使えるようになった」という型で書かれていれば因果は十分とみなし、その因果の詳細さや論理性（なぜそうなるかの物理・魔術的メカニズム）や「もっと明確に」を要求して指摘してはならない。
- logline が物語の方向性（静かな日常、内面的な葛藤、新たな決断等）を示す表現を含む場合、Concept段階ではこれ以上具体化を強制しない。後続工程で具体的に描写されるため。「誰が何をどうするか」の大枠（主人公、危機/葛藤、行動目標）があれば十分。
- `after` は既存設定の最小修正に限定する。新しいキーアイテム、解除条件、制度名、用語、対立構造、因果関係を追加して別設定に作り替えない。
- `world_rules` 同士が同じ中核ギミックを別角度から説明しているだけなら重複・矛盾扱いしない。対象、条件、効果、期限が同時に両立不能な場合だけ指摘する。
- ある内容が既に他のフィールド（別の `world_rules` 要素、`logline`、`world_summary`、`selling_points` 等）で明記されている場合、それを別のフィールドにも書くことを要求してはならない。JSON 全体を総合的に読み、既に書かれている内容の重複要求や、満たされている要件への「もっと詳しく」は指摘しない。
- 自然なカタカナ語、英語表記、英字略語、一般的なジャンル語、固有名詞、日本語として成立する漢語は言語純度の問題にしない。
- `world_rules`・`selling_points` に「〜を描く」「〜を示す」「読者は」「追体験する」などの作品外の執筆方針・読者向けメタ説明が含まれる場合は指摘する。
- 出版可否、総評、長所、スコア、任意の好みは出力しない。
- `issues` が空配列なら改訂不要、1件以上なら改訂を継続する。
- `issues` は最大8件に限定し、後続工程を止めるimportant な問題を1回のレビューでまとめて指摘する。
- `description` と `suggestion` は短文にする。代替案の列挙、比較、検討過程、二重引用符を書かない。
- `after` は1つの完成値だけを書く。複数案や補足説明を混ぜない。

### 指摘の書き方
- 問題がない場合は、無理に指摘を作らず `issues` を空配列にする。
- 指摘する場合は、対象フィールド、問題、改善方針、修正前後の差分が JSON 単体で理解できるように書く。
- `before` と `after` は必ず文字列にする。欠落フィールドの `before` は空文字列、`after` は即採用可能な具体案にする。
- `before` には入力JSON内の実際の値だけを書く。`タイトル:`、`ターゲット読者:`、`章設計:` などの説明ラベルやMarkdown見出しを混ぜない。
- `after` に「〇〇」「△△」「XX」「（例）」などのプレースホルダーを残さない。
- 自然なカタカナ語、英語表記、英字略語、一般的なジャンル語、固有名詞、日本語として成立する漢語は問題にしない。日本語文脈で不自然な簡体字、中国語構文、英語混在、ハングル混在は、問題文字列を引用して指摘する。
- Concept段階では、人物の詳細な職能、関係性の細部、具体的な作戦手順、専門技術、場面単位の実行方法は Character/Design 工程で具体化できるため、企画の中核矛盾でない限りimportant 指摘にしない。
