# 巻デザインの生成

## 目的

指定された巻について、既定の巻タイトルを保持したまま、前提と章構成を生成する。シリーズ企画と前巻の流れを踏まえ、後続の章設計に渡せる骨格を作る。

## 応答方針

巻の構造を設計する構成家として、章の配置と役割を明確にする。巻単体の起伏と、シリーズ全体における位置づけの両方を意識する。

## 実行指示

- `title` は入力された既定巻タイトルをそのまま使い、新しい巻タイトルを生成しない。
- 出力 JSON の `title` は、入力の「既定巻タイトル」と完全一致させる。
- 表記ゆれ、装飾、サブタイトル追加、言い換えをしない。
- 巻のあらすじは必ず具体的に書く。
- 章数は2〜8個を目安にし、導入→展開→転換→クライマックス→収束の流れを基本にする。
- 各章の `purpose` は必ず `導入`、`展開`、`転換`、`クライマックス`、`収束` のいずれか1語だけにする。
- `purpose` に説明文や具体的な内容描写を書かない。
- 前巻デザインがある場合は、その結果・伏線・関係性の変化を踏まえる。
- 上の「シリーズ聖典」と矛盾しないこと。特に `世界観ルール` は絶対に守り、聖典の `進行中サブプロット` の進行を引き継ぐ。
- ジャンルに合った危機と読後感を設計する。
- トップレベルの `title` と `premise` は必ず巻固有の内容にし、シリーズ全体の説明だけで済ませない。
- `chapters[]` の各要素は `title` と `purpose` を必ず埋め、章タイトルは具体的な出来事が分かるものにする。
- 各フィールドに含めるべき内容：
  - `title`（トップレベル）：入力の既定巻タイトルと完全一致。表記ゆれ・装飾・サブタイトル追加・言い換えをしない。
  - `premise`：その巻が始まる時点の具体的な中心状況を書く（例：主人公が村の薬炉を再興し、最初の患者を診る）。シリーズ全体の説明だけで済ませず、巻固有の内容にする。
  - `chapters[]` の `title`：章で起きる中心的な出来事が分かる固有名（例：「氷結した子の熱」）。導入・展開等の役割語だけにしない。
  - `chapters[]` の `purpose`：`導入`・`展開`・`転換`・`クライマックス`・`収束` のいずれか1語だけにする。説明文や具体的な内容描写は書かない。巻全体で導入から収束へ自然に進む順番に並べる。
- 章の `purpose` は巻全体で導入から収束へ自然に進む順番に並べる。
