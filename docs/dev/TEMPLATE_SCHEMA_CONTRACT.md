# テンプレートとスキーマの役割分担

## 基本思想

生成の品質は**プロンプト（テンプレート）の記述**で制御する。
スキーマはあくまで**構造定義**であり、過度な制約を置かない。

```
プロンプト（理想的な出力例）→ LLM が出力 → スキーマ（構造チェック）→ レビュー（品質チェック）
```

## 原則

| | テンプレート (`.md`) | スキーマ (`.json`) |
|---|---|---|
| **役割** | LLM への指示・プロンプト | 構造定義（フィールド名・型・関係） |
| **記述内容** | **理想的な出力の方向性** | `properties`, `required`, `type` |
| **制約** | ✅ **理想値を自然言語で** | ❌ `maxLength`, `maxItems` 等は書かない |
| **期待外れ** | レビューで指摘・修正 | バリデーション失敗のみエラー |

## 詳細ルール

### テンプレートに書くもの

- フィールドの**理想値**（「具体的な危機を描く」「2文以内で」等）
- フィールド名と構造の例（値はプレースホルダ）
- LLM への指示文（「何を書くべきか」）
- 出力形式の注意点（「JSON のみを出力すること」等）

### テンプレートに書かないもの

- `maxLength`, `maxItems` 等の制約（レビューに任せる）
- `required` の有無（スキーマに任せる）
- `enum` の列挙（スキーマに書く場合は `description` で意味を補足）

### スキーマに書くもの

- `type`, `properties`, `required` — 構造定義
- `enum` — 選択肢が有限の場合のみ
- `description` — フィールドの意味（制約は書かない）

### スキーマに書かないもの

- `maxLength`, `minLength` — 削除
- `maxItems`, `minItems` — 削除
- `minimum`, `maximum` — 削除
- レビューの指カテゴリや制約（レビュープロンプトの責務）
- プロンプトの指示文（テンプレートの責務）

## description の書き方

`description` に制約を書く場合は「想定値」として記載する。

```json
{
  "description": "章のテーマ（想定: 300文字）"
}
```

個数制限の範囲も自然言語で書く。

```json
{
  "description": "ジャンル（数個、各想定: 64文字）"
}
```

## 項目数（レビューガイドライン）

章数・シーン数など、**構造的な個数**は LLM の判断に委ねるが、レビューの指摘基準として `description` に範囲を書く。

### 書く場所

| 項目 | 書く場所 |
|---|---|
| 章数 | `volume_design.chapters.description` |
| シーン数 | `volume_design.chapters.scenes.description` |
| キャラクター数 | `series_plan_characters.main_characters.description` |

### 書く内容

バリデータではなく、**レビューで指摘するためのガイドライン**として書く。

```json
{
  "chapters": {
    "type": "array",
    "description": "章のリスト。1巻あたり4-8章が標準。3章以下は短すぎ、10章以上は長すぎ。"
  }
}
```

```json
{
  "scenes": {
    "type": "array",
    "description": "シーンのリスト。1章あたり2-4シーンが標準。"
  }
}
```

### 注意

- `maxLength` は書かない（バリデーションに使わない）
- 例外: レビューschemaの `issues.maxItems` は、1回のレビューで扱う指摘数を制御するために許可する
- プロンプトには書かない（レビューの責務）
- レビューがこの範囲を基準に指摘する

## 期待外れの処理

| 箇所 | 処理 |
|---|---|
| テンプレートの理想値を超える | **レビューで指摘** → revision で修正 |
| スキーマの構造違反（必須フィールド欠落等） | バリデーションエラー → リトジ |
| レビューも修正できない | プロンプト自体の改善 |

## チェックリスト

テンプレートを修正する際に確認すること:

- [ ] テンプレートに理想値が記載されているか？
- [ ] テンプレートに余計な制約を書いていないか？
- [ ] スキーマから `maxLength`/`maxItems` を削除したか？
- [ ] スキーマの `description` に「想定値」のみ記載されているか？
- [ ] 章数・シーン数のレビューガイドラインが `description` に書かれているか？

スキーマを修正する際に確認すること:

- [ ] 構造（properties, required）は正しいか？
- [ ] `maxLength`/`maxItems`/`minItems` 等が残っていないか？
- [ ] `description` に制約ではなく想定値が書かれているか？
- [ ] 項目数（章数・シーン数）のレビューガイドラインが書かれているか？

## 理由

1. **プロンプトが品質を制御**: LLM の出力はプロンプトの指示に最も強く影響される。数値制約よりも「何を期待しているか」の方が効果的。

2. **レビューで柔軟に対応**: 厳しい制約をスキーマに置くと、LLM が正しい出力をしてもバリデーションエラーになる。レビューで指摘する方が柔軟で品質が向上する。

3. **メンテナンス性**: 制約を変更する場合、スキーマではなくプロンプトを修正するだけで済む。

4. **LLM の特性**: `description` に自然言語で「想定: 300文字」と書けば、LLM は概ねそれに従う。従わない場合はレビューで指摘する。
